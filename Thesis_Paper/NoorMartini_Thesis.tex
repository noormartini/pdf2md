\section{Introduction}
% ================================================================

The rapid expansion of digital information has fundamentally changed how
organizations store, share, and process documents. The Portable Document Format
(PDF) has become the de facto standard for document exchange across industries
including law, finance, healthcare, academia, and government. PDF remains the dominant
archival and distribution format across industries, and its continued dominance
makes automated processing an increasingly critical challenge. Despite this ubiquity, the PDF format presents
a fundamental paradox: it excels at preserving visual fidelity for human readers
while simultaneously obstructing automated, machine-readable access to the
information it contains \cite{whitington2012}.

Existing approaches to PDF-to-Markdown conversion fall broadly into three
categories. Traditional text-based parsers such as PyMuPDF and pdfplumber \cite{adhikari2024} extract raw text from native PDFs by
reading embedded glyph data directly, but discard layout structure in the
process. Optical Character Recognition (OCR)-based tools such as Tesseract \cite{smith2007} address scanned
documents by treating pages as flat images and recognizing characters pixel by
pixel, recovering characters without semantic hierarchy. A newer generation of
AI-powered systems, including Marker, Microsoft
MarkItDown, LlamaParse, and multimodal OCR models such as Mistral OCR
\cite{parsebench2026}, applies deep
learning to recover more structure, handling tables, formulas, and multi-column
layouts with greater fidelity. A common pipeline combines these stages: scanned
pages are first processed with an OCR engine, then passed to a layout-aware
conversion tool, and finally cleaned with regex-based post-processing.

Despite these advances, a fundamental limitation persists across all current
tools: each applies a single, fixed extraction strategy to the entire document.
A tool optimized for text-heavy academic papers will apply the same logic to a
page containing only a scanned diagram or a dense mathematical formula. This
one-size-fits-all assumption breaks down on real-world documents, which
routinely mix plain text, tables, images, source code, and equations within the
same file --- and sometimes within the same page. No existing open-source tool
selects its extraction strategy adaptively based on what each page actually
contains.

This thesis addresses this paradox by developing and evaluating a multimodal,
content-aware system that converts PDF documents into structured Markdown. Unlike
existing tools that apply a single fixed extraction strategy to an entire
document, the proposed system analyzes each page individually and selects the
most appropriate strategy based on the detected content type: plain text,
images, tables, hyperlinks, page numbers, source code, or mathematical formulas.
This per-page adaptive approach is powered by LM Studio, a local LLM runtime,
ensuring full reproducibility without dependence on external cloud services. The
conversion from PDF to Markdown is not merely a format transformation; it is a
semantic reconstruction: recovering headings, lists, tables, links, code blocks,
and document hierarchy from a format that encodes position and appearance, but
not meaning. This chapter presents the problem motivating this research, the
objectives driving the design of the proposed system, and the scope within which
the work is situated.

\subsection{Problem Statement}
% ----------------------------------------------------------------

\subsubsection*{Difficulty of Extracting Structured Content from PDFs}

PDF was designed to faithfully reproduce the visual appearance of a printed
page, not to encode its meaning. As a result, the format provides no native
representation of semantic elements such as paragraphs, headings, list items,
or table cells \cite{whitington2012}. Any structure visible to a human reader
must be reverse-engineered by software from low-level positional and typographic
signals, a process that is inherently fragile and error-prone
\cite{parsing2024}. Real-world documents compound this difficulty by mixing
plain text, embedded images, mathematical formulas, multi-column layouts, and
complex tables within the same file, meaning that a single extraction strategy
applied uniformly across all pages will inevitably fail on some content types.
The technical details of the PDF format and the specific challenges each content
type poses are covered in Chapter~2.

\subsubsection*{Limited OCR Accuracy for Complex Layouts}

Optical Character Recognition (OCR) is the classical solution for image-based
PDFs — scanned documents where text exists only as pixels. However, OCR accuracy
degrades sharply when documents contain complex layouts. Subramani et al.\
\cite{subramani2020} identify document layout analysis as a prerequisite step: a
model must first identify and categorize content regions (text blocks, figures,
tables, equations) before OCR can be meaningfully applied. Traditional OCR
pipelines, including widely-used tools such as Tesseract~\cite{smith2007}, treat
the document page as a flat image and struggle to infer the semantic role of
detected text regions. This limitation is quantitatively demonstrated by Zhu et
al.\ \cite{zhu2022}, who show that applying OCR directly to complex documents
without prior layout segmentation yields a normalized edit distance of 0.63,
whereas adding a layout segmentation precursor reduces it to 0.38 (lower is
better) --- a substantial improvement achieved solely by first decomposing the
page into semantically coherent regions.

A primary driver of this degradation is multi-column layout: articles spanning
multiple columns cause OCR engines to merge or mis-order text across columns,
violating reading order, and Zhu et al.\ \cite{zhu2022} demonstrate that
preserving human read-order in such documents requires a dedicated segmentation
stage before any character recognition is attempted. Modern surveys and
comparative studies confirm that even state-of-the-art OCR models face challenges
with dense multi-column text, distorted or rotated text, tabular content
requiring row-column alignment, and mathematical notation
\cite{subramani2020, adhikari2024}. In a study of end-to-end
PDF-to-Markdown conversion, Blecher et al.\ \cite{blecher2023} note that the PDF
format causes ``a loss of semantic information, particularly for mathematical
expressions,'' a point reinforced by Darvishy~\cite{darvishy2018}, who observes
that no existing tool provides mechanisms to interpret or describe mathematical
formulas and scientific graphs. This underscores that the limitation is not
purely a pixel-level recognition problem but an inherently structural one ---
and directly motivates the content-aware, per-page strategy developed in this
thesis.

\subsubsection*{Need for Conversion to Markdown for Accessibility and Version Control}

The need to convert documents to Markdown arises from a convergence of practical
demands across software development, scientific publishing, and artificial
intelligence. Markdown is a lightweight markup language created by John Gruber in 2004,
designed to be readable in raw form while converting cleanly to HTML and other
structured formats \cite{cone2019}. Cassel et al.\ \cite{cassel2025} characterize it as ideal
for machine-readability due to its simple syntax and structured format, noting
that community extensions further support tables and advanced mathematical
notation, making it a well-suited target for document conversion pipelines.

Three properties make Markdown especially valuable as a target representation:

\begin{enumerate}
  \item \textbf{Version control compatibility.} Because Markdown is plain text,
        every change is tracked, attributed, and auditable in version control
        systems such as Git, treating a document as a fully versioned project
        \cite{saraiva2025}.

  \item \textbf{Machine-readability and structured format.} Markdown's simple
        syntax and structured format preserve context and enhance comprehensibility
        for both humans and automated systems, with community extensions further
        supporting tables and advanced mathematical notation \cite{cassel2025}.

  \item \textbf{Portability and long-term accessibility.} Markdown files can be
        opened with virtually any text editor on any operating system, with no
        dependency on proprietary software. Even if a specific application stops
        working, the content remains fully readable --- an important consideration
        for documents that need to be preserved indefinitely \cite{cone2019}.

  \item \textbf{LLM and RAG pipeline integration.} The growth of large language
        models (LLMs) and retrieval-augmented generation (RAG) architectures has
        created strong demand for structured plain-text document representations.
        Structured Markdown preserves semantic hierarchy --- headings, lists,
        tables --- that improves retrieval precision and context quality when
        documents are ingested into AI pipelines \cite{khan2024}.
\end{enumerate}

\subsection{Motivation}
% ----------------------------------------------------------------

\subsubsection*{Growing Importance of Document Digitization}

The global drive toward document digitization is accelerating across all sectors.
Increasing digital transformation investments and the proliferation of AI-driven
workflows are creating strong demand for structured and searchable document storage.
Organizations are no longer merely scanning documents but are
increasingly requiring semantic understanding: the ability to extract, query,
and transform document content programmatically. PDF-to-Markdown conversion sits
at the intersection of this shift, serving as the bridge between static archival
documents and the dynamic, queryable knowledge bases that modern workflows demand.

\subsubsection*{Advantages of Markdown as a Structured Representation}

Markdown offers a unique balance between human readability and machine
processability. Unlike proprietary formats such as Word or PDF, Markdown does
not rely on closed specifications and can be used to create websites, documents,
books, presentations, and technical documentation, and converted between these
formats using tools such as Pandoc. Its widespread adoption
across platforms such as GitHub and Reddit means Markdown files can be opened
with virtually any application on any operating system, and remain readable even
if the software used to create them is no longer available. This flexibility allows
document content to be \textit{written once and adapted for various platforms
without reformatting} \cite{cone2019}.

\subsubsection*{Potential of LLMs in Understanding Context and Structure}

The emergence of large language models with vision capabilities has opened new
possibilities for document understanding that transcend traditional OCR-based
approaches. Blecher et al.\ \cite{blecher2023} introduced Nougat (Neural Optical
Understanding for Academic Documents), a Visual Transformer model that converts
images of PDF pages directly into a structured markup language. Their key insight
is that an end-to-end model trained on paired PDF-markup data can infer semantic
structure (headings, equations, tables, and citations) without requiring an
explicit layout analysis step. This demonstrated the viability of replacing
multi-stage OCR pipelines with a single multimodal model.

Building on this direction, Wang et al.\ \cite{wang2024} developed MinerU, an
open-source dual-engine system combining Vision-Language Models (VLMs) and OCR
to convert PDFs and other document types into structured Markdown and JSON,
supporting 109 languages and automatic detection of scanned versus
digitally-born PDFs. More recent work by Cassel et al.\ \cite{cassel2025}
demonstrates that end-to-end decoder transformer models outperform pipeline-based
methods on accuracy metrics while offering greater flexibility on diverse document
types.

Complementing these document-level findings, Cheng et al.\ \cite{cheng2024}
demonstrate that LLMs can capture structured semantic representations via
prompting alone, without task-specific fine-tuning, though limitations emerge
when handling complex argument structures. This supports the theoretical
basis for using a vision LLM to recover structured Markdown from rendered
page images, while also setting bounded expectations for content types such
as dense mathematical formulas.

These developments motivate the central premise of this thesis: that a
\textit{hybrid} extraction strategy combining text-layer extraction with
image-based LLM understanding can achieve accuracy and structural fidelity
that neither modality achieves alone.

inefficiency of applying a fixed strategy uniformly. The adaptive approach
addresses this by routing each page to the most appropriate strategy based on

\subsection{Research Objectives and Questions}
% ----------------------------------------------------------------

Given the identified problem and motivation, this thesis pursues the following
primary research objective:

\begin{quote}
\textit{To design, implement, and evaluate a multimodal, content-aware system
that converts PDF documents into structured Markdown by selecting an extraction
strategy per page based on detected content type, powered by LM Studio as a
local LLM runtime to ensure full reproducibility without dependence on external
cloud services, and to assess whether this adaptive approach outperforms fixed
text-only, image-only, and hybrid strategies with respect to accuracy,
structural fidelity, and computational efficiency.}
\end{quote}

The system achieves this by analysing each PDF page individually and
classifying it by content type before conversion. Pages identified as
plain text are converted immediately using PyMuPDF without any LLM
involvement, keeping processing fast and deterministic. Pages that
contain visual or structured content --- images, diagrams, mathematical
formulas, tables, source code, headings embedded in layouts, or
hyperlinks --- are rendered as images and sent to LM Studio, where a
locally-hosted vision-capable LLM performs the conversion using a
prompt tailored to the detected content type. This routing decision is
made per page, so a single document can combine both pathways depending
on what each page contains.

This objective is operationalized through the following research questions:

\begin{description}
  \item[RQ1.] \textit{Adaptivity:} Does a content-aware system that selects an
        extraction strategy per page based on detected content type (plain text,
        image, table, source code, formula, etc.) achieve higher accuracy and
        structural fidelity than any single fixed strategy applied to the full
        document?

  \item[RQ2.] \textit{Strategy comparison:} How do text-only, image-only, and
        hybrid extraction strategies individually compare in terms of content and
        structural accuracy across different PDF document categories?

  \item[RQ3.] \textit{Efficiency:} What are the computational costs
        (processing time, token usage, and number of LLM calls per document)
        of each strategy and of the adaptive system, and how do these trade
        off against accuracy gains?

  \item[RQ4.] \textit{Output quality:} How can the quality of generated
        Markdown be reliably assessed beyond standard n-gram metrics (BLEU,
        ROUGE), given that structured output requires evaluating heading
        hierarchy, list fidelity, and table reconstruction?

  \item[RQ5.] \textit{Generalizability:} Do the observed performance
        differences between strategies vary systematically across document types
        (scientific papers, forms, presentations, scanned documents)?
\end{description}

These questions are guided by the following hypotheses, which are evaluated
against the experimental results in Chapter~6.

\begin{description}
  \item[H1 — Adaptive superiority.] The adaptive system will achieve higher
        overall structural fidelity and content accuracy than any single fixed
        strategy, because different page types have fundamentally different
        extraction requirements that no one strategy handles equally well.

  \item[H2 — Strategy-content fit.] Text-only extraction will perform best on
        plain-text pages but degrade on image-heavy or formula-rich pages.
        Image-only extraction will show the opposite profile: it renders every
        page as an image and processes it through the vision LLM regardless of
        content type, meaning plain-text pages are penalised by unnecessary
        visual rendering and LLM inference, which introduces noise compared to
        direct text extraction.

  \item[H3 — Computational efficiency of the adaptive approach.] Despite
        classifying each page before conversion, the adaptive system will have a
        lower average computational cost than the image-only or hybrid
        strategies, because plain-text pages are handled by PyMuPDF locally
        without any LLM call, and non-text pages are processed with prompts
        tailored to their content type rather than a single general-purpose
        prompt, reducing wasted inference on irrelevant instructions.

  \item[H4 — Hybrid as upper bound for mixed content.] For documents with a
        balanced mix of text and visual content, the hybrid strategy will serve
        as a strong baseline that the adaptive system matches or exceeds by
        avoiding unnecessary LLM calls on purely textual pages.
\end{description}

\subsection{Scope and Delimitations}
% ----------------------------------------------------------------

This thesis focuses on PDF documents as the exclusive input format. The
developed system is primarily evaluated on German and English academic
documents, with the experimental benchmark covering four representative
document categories: academic papers, administrative forms, presentation
slides, and scanned archival documents. While the system architecture is
designed to be compatible with any LLM backend supporting a vision API, all
experiments are conducted locally using LM Studio to ensure full
reproducibility without dependence on external cloud services.

The system performs automatic language detection using the \texttt{langdetect}
library and adjusts LLM prompts accordingly, appending a language-preservation
instruction for non-English documents to prevent the model from translating
content. However, no language-specific training or fine-tuning is applied, and
the evaluation metrics are language-agnostic. The primary experimental results
and qualitative analysis are based on the German-language test document.

The page classification module identifies five content types (TEXT, IMAGE,
FORMULA, MIXED, and EMPTY) based on structural signals extracted by
PyMuPDF: embedded image count, vector path density, presence of mathematical
Unicode symbols, and total character count. Font-size-based heading detection
is applied to text pages using a character-weighted font-size tally to identify
the body text size and promote larger spans to Markdown heading levels H1--H3.

The following aspects are explicitly out of scope:

\begin{itemize}
  \item Real-time or streaming document processing.
  \item Conversion of formats other than PDF (e.g., DOCX, HTML, PPTX).
  \item Fine-tuning or task-specific training of LLM models.
  \item Enterprise-scale deployment or cloud infrastructure integration.
  \item Support for right-to-left scripts or ideographic writing systems.
\end{itemize}

The evaluation focuses on Markdown as a single target format. Conversion to
other structured formats (LaTeX, HTML, JSON) is not addressed, though the
architecture supports such extensions through prompt modification.

\subsection{Thesis Contributions}
% ----------------------------------------------------------------

This thesis makes the following contributions to the field of document
processing and multimodal AI systems:

\begin{enumerate}
  \item \textbf{Per-page content classifier.} A lightweight page classification
        module that assigns each PDF page to one of five content types
        (TEXT, IMAGE, FORMULA, MIXED, or EMPTY) using structural signals
        extracted by PyMuPDF: embedded image count, vector path density,
        mathematical symbol detection, and character count. The classifier
        operates without any LLM call, keeping classification fast and
        deterministic.

  \item \textbf{Font-size-based heading detection.} A heading detection
        mechanism that uses a character-weighted font-size tally to identify
        the dominant body text size on each page and promotes spans exceeding
        this baseline by thresholds of 15\%, 40\%, and 80\% to Markdown heading
        levels H3, H2, and H1 respectively. This replaces flat text extraction
        with structure-preserving Markdown output for text-classified pages.

  \item \textbf{Per-page figure extraction with Markdown linking.} An image
        extraction pipeline that retrieves embedded figures from vision-processed
        pages using PyMuPDF's \texttt{extract\_image} API, saves them as
        individual files (e.g.\ \texttt{figures/page\_001\_fig\_001.png}), and
        instructs the vision LLM to reference these paths as Markdown image
        links in its output, preserving the visual content of non-text pages.

  \item \textbf{Seven-metric evaluation framework.} A structured evaluation
        methodology that assesses Markdown output quality across seven
        complementary metrics: text similarity (difflib SequenceMatcher ratio),
        heading structure score, list structure score, table structure score,
        code block score, paragraph structure score, and word overlap (Jaccard
        similarity). This multi-dimensional approach captures aspects of
        structural fidelity that standard n-gram metrics such as BLEU and ROUGE
        cannot measure.

  \item \textbf{Systematic experiment runner.} A configurable experiment
        framework that evaluates all combinations of strategy, model,
        prompt variant, temperature, and page across a benchmark of reference
        documents, records per-page scores and token usage, and generates
        summary reports for strategy and model comparison.

  \item \textbf{Reproducible open-source implementation.} A fully documented
        implementation using open-source tools (PyMuPDF, pymupdf4llm, LM Studio)
        that runs entirely locally, with all prompt templates, evaluation scripts,
        experiment configurations, and test suites included.
\end{enumerate}

\subsection{Thesis Structure}

The remainder of this thesis is organized as follows:

\textbf{Chapter 2 (Background and Fundamentals)} provides the theoretical
foundation for this work. It covers the PDF internal structure and its
parsing challenges, Markdown syntax and its role as a structured target
representation, the architecture and vision capabilities of modern large
language models, and a taxonomy of multimodal extraction strategies.

\textbf{Chapter 3 (Related Work)} surveys existing tools and research across
three areas: traditional PDF processing tools (PyMuPDF, pdfplumber, Tesseract),
modern conversion systems (Marker, LlamaParse, Nougat, MinerU), and LLM-based
document understanding. It identifies the limitations that motivate the
adaptive approach developed in this thesis.

\textbf{Chapter 4 (Methodology)} describes the research design and
comparative study framework, the document dataset and ground truth construction
process, and the seven-metric evaluation methodology used to assess Markdown
output quality. It defines the experimental conditions under which all four
strategies are compared.

\textbf{Chapter 5 (System Design and Implementation)} presents the full
architecture of the developed system: the per-page content classifier, the
font-size-based heading detection mechanism, the figure extraction and Markdown
linking pipeline, the four conversion strategies and their LM Studio
integration, the content-type-specific prompt designs, and the post-processing
pipeline. Implementation decisions and the technology stack are documented here.

\textbf{Chapter 6 (Results and Evaluation)} reports the experimental
outcomes, including per-metric scores by strategy, model, temperature, and
document type; qualitative case studies of success and failure modes; and a
strategy comparison summary with recommended use cases for each approach.

\textbf{Chapter 7 (Discussion)} interprets the results in light of the five
research questions and four hypotheses, addresses threats to internal and
external validity, and situates the findings within the broader landscape of
document understanding research.

\textbf{Chapter 8 (Conclusion)} summarizes the contributions and key
findings, acknowledges the limitations of the study, and outlines directions
for future work including multilingual evaluation, support for additional
document formats, and integration with newer multimodal LLM architectures.

\subsection{PDF Format and Parsing Challenges}
\begin{itemize}
    \item PDF Internal Structure
    \begin{itemize}
        \item \textbf{PostScript heritage:} PDF evolved from PostScript, a page
              description language designed for rendering, not for storing structured
              information --- this origin explains why PDF encodes content as drawing
              instructions rather than semantic elements \cite{whitington2012}
        \item PDFs as ``drawing instructions'': text stored as glyphs at $(x, y)$
              coordinates with no native concept of paragraph, heading, or list ---
              all semantic structure must be reverse-engineered from geometric
              proximity, font size, and alignment cues \cite{parsing2024}
        \item \textbf{PDF coordinate system:} the page origin is at the bottom-left
              corner with $Y$ increasing upward, opposite to screen coordinates ---
              positional sorting requires coordinate transformation and still fails
              on multi-column layouts \cite{whitington2012}
        \item \textbf{Tagged vs.\ untagged PDFs:} ISO 32000-1 version 1.4
              introduced ``tagged PDF,'' adding a logical structure tree that maps
              rendering commands to semantic roles such as headings and paragraphs
              and specifies reading order \cite{darvishy2018}; the vast majority of
              PDFs are untagged, meaning structure must be inferred purely from
              geometry, and even when tags exist, correct reading order still
              requires manual verification \cite{darvishy2018}
        \item Distinction between \textbf{native PDFs} (contain actual text objects
              with bounding boxes, amenable to structural parsing) and
              \textbf{scanned PDFs} (image-only pages requiring OCR and visual
              region detection before any text can be extracted) \cite{parsing2024}
    \end{itemize}
    \item Text Extraction Challenges (embedded fonts, encoding, layouts)
    \begin{itemize}
        \item \textbf{Font encoding complexity:} PDF can embed fonts with custom
              encoding vectors, meaning character codes in the file do not directly
              map to Unicode --- a root cause of garbled or incomplete text
              extraction, especially in older documents \cite{whitington2012}
        \item Flattened text extraction: loses paragraph boundaries, list
              structure, and semantic grouping \cite{adhikari2024}
        \item Over-applying OCR to native PDFs: introduces noise and discards
              existing formatting metadata \cite{adhikari2024}
        \item Multi-column layouts, footnotes, and mixed-language documents
              violate the assumption that positional order equals reading order
    \end{itemize}
    \item Visual/Image-based Extraction Challenges
    \begin{itemize}
        \item Images extracted without positional context lose their semantic
              relationship to surrounding text \cite{parsing2024}
        \item Tables: PDF encodes no explicit row/column structure; complex
              layouts may require HTML fallback since Markdown lacks
              \texttt{rowspan}/\texttt{colspan} support \cite{parsing2024}
        \item Code blocks: identifiable through monospaced font detection and
              consistent horizontal spacing, but easily fragmented by standard
              text extractors \cite{adhikari2024}
    \end{itemize}
\end{itemize}
